## Phases

### Phase 06 ‚Äî Observability Dashboard
**Goal:** Users can inspect live systemd journal output and VM vitals (disk, RAM, service status) directly from the web UI without ever opening an SSH session.

**Requirements:** LOGS-01, LOGS-02, LOGS-03, HEALTH-01, HEALTH-02, HEALTH-03

**Plans:**
- `06-01-PLAN.md` ‚Äî Host Vitals widgets on main dashboard (HEALTH-01..03)
- `06-02-PLAN.md` ‚Äî Systemd Journal viewer as third tab on `/logs` (LOGS-01..03)

**Key technical decisions:**
- Zero new dependencies (Node built-ins + Debian binaries only)
- `MemAvailable` from `/proc/meminfo`, not `os.freemem()`
- SSE with ReadableStream for live log tail; abort-handler + cancel both kill the child process to prevent orphan `journalctl -f` zombies
- `execFile` with argv arrays (no shell injection)

**VM verification:** PASSED ‚Äî 3 manual checks (SSE orphan cleanup, live propagation, service state) all green.

### Phase 07 ‚Äî Backup & Restore
**Goal:** Users can download and restore the SQLite database from the UI as a safety net before risky operations.

**Requirements:** BACKUP-01, BACKUP-02, BACKUP-03

**Plans:**
- `07-01-PLAN.md` ‚Äî Backup service + download/restore API + settings Backup tab with confirmation modal

**Key technical decisions:**
- Uses `better-sqlite3.backup()` for online backup with WAL checkpointing
- Atomic `fs.renameSync()` for restore (POSIX-atomic)
- `PRAGMA integrity_check` + required-tables validation before replacing live DB

- Confirmation gate via `confirmed: true` body field

**VM verification:** PASSED ‚Äî live download (45 KB valid SQLite file, 7 tables, integrity ok), confirmation gate (HTTP 400 without flag), integrity rejection (HTTP 400 for garbage).

### Phase 09 ‚Äî Update Runner & Rollback
**Goal:** Users can trigger a self-update from the UI with live log streaming, safety guards, and automatic rollback on failure.

**Requirements:** UPDATE-04, UPDATE-05, UPDATE-06, UPDATE-07, UPDATE-08

**Plans:**

**Key technical decisions:**

## Key Accomplishments

1. **Zero-SSH operations:** Self-hosters can now observe, back up, and update the running app entirely from the web UI. No `ssh root@host` needed for routine maintenance.

2. **Professional self-update mechanism:** Detached systemd-run unit, live SSE log streaming, automatic rollback, schema-change warning, and safety guards (dirty tree, up-to-date check). The feature **successfully updated itself** with its own history-persistence fix during verification ‚Äî the ultimate dogfood test.

3. **Production-hardened SIGTERM handling:** Discovered during live verification that the Node server was ignoring SIGTERM, causing every `systemctl restart` to take 90 seconds. Added signal handlers in `hooks.server.ts` so restarts complete in ~3 seconds.

4. **Observability without Prometheus:** Host vitals (disk/RAM/service) via `/proc/meminfo` + `df` + `systemctl show`, and live journal tail via SSE, all with zero new dependencies. Piggybacks on the existing 10s dashboard poll loop.

5. **Rollback-by-default philosophy:** Every risky operation has an escape hatch. Backup before restore. git reset on build failure. Reconcile orphaned entries on boot. Dirty-state blocks updates. The user never ends up in an inconsistent state.

## Decisions Logged

- **Rolling `main` updates** (not tag-based releases) ‚Äî simpler, no tag discipline required, matches the single-user homelab audience
- **Schema-hash warning as stopgap** for schema migrations ‚Äî real Drizzle migration system is a future requirement
- **Daily auto-check + badge** (not in-layout live polling) ‚Äî respects GitHub API rate limits with multiple tabs open
- **File-based SSE streaming** (not in-memory pipe) ‚Äî survives the app's own restart, which is essential for self-updates
- **Git pull over rsync** as the canonical deploy mechanism for end users ‚Äî the one-time `git reset --hard origin/main` migration is documented in Phase 09 SUMMARY
